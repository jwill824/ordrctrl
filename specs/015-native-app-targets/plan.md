# Implementation Plan: Native App Targets (Capacitor + Tauri)

**Branch**: `015-native-app-targets` | **Date**: 2026-03-14 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/015-native-app-targets/spec.md`

## Summary

Wrap the existing Vite SPA in **Capacitor 6** for iOS/Android distribution (TestFlight + internal Android testing) and in **Tauri 2** for macOS/Windows distribution (internal/direct download). Both targets use the existing React/TypeScript frontend without modification — native platform scaffolding lives alongside the web source. Local notifications are triggered from the frontend's feed-polling cycle using platform-specific plugins behind a shared abstraction. The backend requires targeted CORS + cookie changes to allow native webview origins.

## Technical Context

**Language/Version**: TypeScript 5.4 (frontend), Rust 1.75+ (Tauri shell), Swift/Kotlin (Capacitor native bridge — generated)
**Primary Dependencies**:
- Capacitor 6 (`@capacitor/core`, `@capacitor/cli`, `@capacitor/ios`, `@capacitor/android`)
- Capacitor plugins: `@capacitor/local-notifications`, `@capacitor/app`, `@capacitor/preferences`
- Tauri 2 (`@tauri-apps/cli`, `@tauri-apps/api`)
- Tauri Rust crates: `tauri-plugin-notification`, `tauri-plugin-deep-link`, `tauri-plugin-single-instance` — tray support is built into the `tauri` crate itself via `features = ["tray-icon"]` (no separate `tauri-plugin-tray` crate exists)
- Tauri npm packages: `@tauri-apps/plugin-deep-link`, `@tauri-apps/plugin-os`, `@tauri-apps/plugin-notification`, `@tauri-apps/plugin-opener`
- `tauri-plugin-opener` Rust crate — opens URLs in the system browser from Tauri (used by OAuth flow)

**Storage**: No new storage layer — session cookies persisted by native WebView OS cookie jar; `@capacitor/preferences` for lightweight native-only flags (e.g., last-seen notification timestamp)
**Testing**: Vitest (existing, unit), Playwright (existing, e2e web), manual device testing for native
**Target Platform**: iOS 14+ (Capacitor requirement), Android 10+ (API 29), macOS 11 Big Sur+, Windows 10+
**Project Type**: Mobile app (iOS/Android) + desktop app (macOS/Windows) wrapping existing web SPA
**Performance Goals**: App cold-start ≤3s on mid-range device; notification delivery ≤1s after trigger
**Constraints**: No new UI surfaces; no new backend endpoints for the native wrapper itself; cookie-based auth must remain unchanged
**Scale/Scope**: Single-user app; same 10-screen SPA wrapped; no new data storage concerns

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

| Principle | Status | Notes |
|-----------|--------|-------|
| **I. Integration Modularity** | ✅ PASS | Native plugin calls are wrapped in `src/plugins/` — isolated from core app logic. Notifications abstracted behind `NotificationService`; deep-link handler isolated in `src/plugins/deep-link.ts`. |
| **II. Minimalism-First** | ✅ PASS | Zero new UI screens. System tray and notification tap navigate to existing routes. No new configuration options exposed to users. |
| **III. Security & Privacy** | ✅ PASS | Session cookies remain `httpOnly` — never in JS storage. No tokens in `localStorage` or Capacitor Preferences. OAuth callbacks handled server-side; only navigation occurs client-side after callback. |
| **IV. Test Coverage** | ✅ PASS | Plugin wrapper modules (`src/plugins/`) have unit tests with vitest mocks. CORS/session changes have integration tests. Native notification scheduling logic is unit-testable via the shared abstraction. |
| **V. Simplicity & Deferred Decisions** | ✅ PASS | Two native wrappers (Capacitor + Tauri) are justified in spec as P1/P2 user stories. Complexity tracking entry added below for the dual-platform approach. |

**Post-Phase 1 re-check**: ✅ No new violations introduced by data model or contracts design.

**Structure Decision**: Native platform scaffolding lives **inside** `frontend/` alongside the Vite source. This keeps the mono-repo structure intact (`backend/` + `frontend/`) and avoids a 3rd project root. The Capacitor and Tauri CLI tools both default to this convention. The `src/plugins/` directory provides the Integration Modularity boundary required by Constitution I.

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|--------------------------------------|
| Two native wrappers (Capacitor + Tauri) | Spec has separate user stories for mobile (P1) and desktop (P2) as distinct distribution channels | A single Capacitor-for-desktop approach is possible but Tauri provides a native macOS/Windows experience with system tray, smaller binary (~5 MB vs 80+ MB), and no Android dependency on a desktop build |
| Backend CORS multi-origin | Native webviews use origins (`capacitor://localhost`, `tauri://localhost`) that differ from `APP_URL` | Cookie-based auth cannot use a wildcard `*` origin when `credentials: include` is required; enumerated origins are the only RFC-compliant option |
